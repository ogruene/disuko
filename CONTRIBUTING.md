# Contributing to Disuko

Thank you for your interest in contributing to **Disuko**!

This project is hosted at the **Eclipse Foundation** and follows the
**Eclipse Development Process (EDP)**, which is based on openness,
transparency, meritocracy, and vendor neutrality.

This document describes the terms, communication channels, and processes
for contributing to this project.

---

## 1. Code of Conduct

This project adheres to the **Eclipse Foundation Community Code of Conduct**.

All participants are expected to interact respectfully and professionally
in all project spaces, including repositories, issue trackers, discussions,
mailing lists, and chats.

➡ https://www.eclipse.org/org/documents/Community_Code_of_Conduct.php

---

## 2. Terms of Use

This project is governed by the **Eclipse Foundation Terms of Use**.

By interacting with this project — including participating in discussions,
reporting issues, or submitting contributions — you agree to comply with
these terms.

➡ https://www.eclipse.org/legal/termsofuse.php

Contributions of code or documentation are additionally subject to the
Eclipse Contributor Agreement (ECA) and the project’s declared license(s).

---

## 3. Project Communication

All project communication is conducted in an open and transparent manner.

The **primary communication channels** for this project are:

- Issues and pull requests in this repository

Project-related discussions and decisions are made publicly through these
channels.

Please avoid private communication with individual maintainers unless
explicitly requested (for example, for security vulnerability reporting).

---

## 4. Eclipse Contributor Agreement (ECA)

Before any contribution can be accepted, you **must**:

1. Create an Eclipse Foundation account
2. Sign the **Eclipse Contributor Agreement (ECA)**

The ECA ensures that contributions can be legally distributed under the
project’s license.

➡ Sign the ECA: https://www.eclipse.org/legal/ECA.php

**Note:** Pull requests from contributors without a signed ECA cannot be merged.

---

## 5. Ways to Contribute

You are welcome to contribute by:

- Reporting bugs
- Proposing enhancements
- Improving documentation
- Submitting pull requests
- Reviewing issues or pull requests
- Participating in technical discussions

All contributions are evaluated on technical merit and project fit.

---

## 6. Contribution Workflow

### 6.1 Reporting Issues

- Use the project’s issue tracker
- Clearly describe the problem or enhancement
- Include reproduction steps, logs, or screenshots when applicable

### 6.2 Submitting Pull Requests

1. Fork the repository
2. Create a feature branch
3. Make your changes
4. Ensure commits are clean and focused
5. Open a pull request against the main development branch

---

## 7. Commit Requirements

### 7.1 Signed-off-by

All commits must include a `Signed-off-by` line:

`Signed-off-by: Your Name <your.email@example.com>`

This certifies that you have the right to submit the contribution under the
project’s license.

---

### 7.2 Licensing and Original Work

By submitting a contribution, you confirm that:

- The contribution is your original work, **or**
- You have the right to submit it under the project license
- Any third-party content is clearly identified and properly licensed

Do not include copied code unless license compatibility has been verified.

---

## 8. Pull Request Review Process

- All pull requests are reviewed by project **committers**
- Reviews focus on:
  - Technical correctness
  - Code quality
  - Architectural consistency
  - Licensing and IP compliance
- Maintainers may request changes before merging

Not all contributions are guaranteed to be accepted.

---

## 9. Becoming a Committer

The Eclipse Foundation uses a **merit-based** governance model.

Contributors who:

- Provide consistent, high-quality contributions
- Demonstrate understanding of the project and Eclipse processes
- Engage constructively with the community

…may be nominated and elected as **committers**.

---

## 10. Security Issues

Please **do not report security vulnerabilities via public issues**.

Instead, follow the responsible disclosure process described in:

➡ `SECURITY.md`
